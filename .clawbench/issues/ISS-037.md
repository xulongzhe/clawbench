## Description
Three related issues in the scheduler:
1. `RemoveTask` is non-atomic — performs multiple DB operations without a transaction; errors on lines 279/282 silently discarded.
2. Race condition in trigger — `HasRunningExecutions` check and `runningExecutions.Store` are not atomic; concurrent triggers can both pass the check.
3. `executeTask` updates task stats using stale `task.Status` loaded at execution start, potentially overwriting user-initiated status changes (e.g., pausing).

## Impact
1. Partial deletes leave orphaned data; 2. Duplicate concurrent executions; 3. Paused task gets silently reactivated when execution completes.

## Suggestion
1. Wrap RemoveTask in a transaction; 2. Use sync.Map.LoadOrStore for atomic check-and-register; 3. Re-read current task status from DB before updating.
