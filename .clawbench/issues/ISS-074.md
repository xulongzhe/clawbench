## Description
tool_use timeout callback captures stale `newBlock` reference after `queue_consume`. When a tool_use block is created and a timeout is set, a subsequent `queue_consume` event may replace `newBlock` before the timeout fires. The timeout callback then operates on the old block reference instead of the current one.

## Impact
Timeout actions (e.g., auto-closing a tool_use block) target the wrong block, leading to UI inconsistencies such as stale loading spinners or incorrect block state.

## Suggestion
Use a ref or store the block ID instead of the block reference in the timeout closure. On timeout, look up the block by ID from the current state rather than using the captured reference. Alternatively, cancel the timeout when `queue_consume` replaces the block.
