## Description

When a client disconnects and immediately reconnects, a race exists between `DisconnectClient` (clearing `sub.conn`) and `Subscribe` (setting `sub.conn` to the new connection) at manager.go:100-116. If `DisconnectClient` runs after `Subscribe`, it nils out the new connection.

## Impact

Race condition on reconnect can cause the new connection to lose event delivery. The subscription appears active but `sub.conn` is nil, so events go to buffer/push instead of the WebSocket. The user sees a stale UI that never updates despite being "connected."

## Suggestion

In `DisconnectClient`, check that the subscription's connection is still the one that was active when the goroutine started. Add a connection identity check (e.g., compare pointer values or use a connection UUID) before clearing conn. Only nil out the conn if it still matches the old connection.
