## Description
SSH direct-tcpip handler parses `HostToConnect` from client but never validates it against the local-only policy. The current implementation hardcodes `127.0.0.1` as the dial target, preventing exploitation. However, the code structure is misleading — the parsed host is silently discarded with no explicit check or comment enforcing the local-only constraint.

## Impact
Currently safe due to hardcoded localhost, but a future refactor using the parsed `HostToConnect` would create an SSRF vulnerability (open proxy through SSH tunnel).

## Suggestion
Add explicit check rejecting non-local `HostToConnect` values with a warning log. This makes the security policy explicit and prevents regression.
