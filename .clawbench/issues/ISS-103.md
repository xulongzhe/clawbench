## Description

Session tokens are generated as `SHA-256(password + "clawbench-salt")` at main.go:410-411. This is deterministic — the same password always produces the same token. The static salt `"clawbench-salt"` is identical across all ClawBench instances.

## Impact

Session hijacking is trivially possible from cookie theft because the token is predictable. The hash is deterministic across restarts, so a stolen cookie remains valid indefinitely. The static salt means all instances share the same token derivation, making it even easier to forge tokens if the password is known.

## Suggestion

Generate session tokens with `crypto/rand`. Store a mapping of random-token -> user (in memory or SQLite). The bcrypt hash is fine for password verification; the session cookie should be an opaque random identifier with no derivable relationship to the password. Rotate tokens on server restart.
