## Description

The SSH server silently replaces any `HostToConnect` with `127.0.0.1` at server.go:385, effectively ignoring the client's requested host without validation or rejection. While hardcoding localhost prevents SSRF, it silently accepts invalid requests.

## Impact

A client requesting `google.com:80` gets silently connected to `127.0.0.1:80`. This is confusing for debugging and violates the principle of rejecting invalid input. While the current hardcoded localhost is safe against SSRF, the lack of validation means invalid requests are silently accepted, and any future change to the forwarding logic could introduce SSRF without this guard being noticed.

## Suggestion

Add explicit validation: if `d.HostToConnect != "127.0.0.1" && d.HostToConnect != "localhost"`, reject the channel with an appropriate error message. This makes the security boundary explicit and auditable.
