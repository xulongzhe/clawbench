## Description
If `executeTask` (scheduler.go:572-579) panics or exits before the deferred cleanup removes the task from `taskRunning`, the task remains permanently stuck as "running" and will never execute again. The deferred cleanup is not guaranteed to run on panic if the panic is not recovered.

## Impact
A single panic during task execution permanently deadlocks that scheduled task. The task will never execute again, even on subsequent cron ticks. Users must manually restart the server to clear the stuck state.

## Suggestion
Add a panic recovery wrapper around `executeTask` that ensures the deferred cleanup always runs. Use `defer` in combination with `recover()` to catch panics and guarantee that `taskRunning.Delete(taskID)` is always called. Also consider adding a startup check that clears stale `taskRunning` entries.
