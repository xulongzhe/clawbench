## Description
The 'error' SSE event handler in `useChatStream.ts` (lines 461-477) calls `onLoadHistory()` which is async. The `.catch()` handler contains a `guard()` check that can fail if the session changes during the async gap. When this happens, `loading.value` is never set to false, leaving the UI in a permanent loading state with no recovery except a page reload.

The same stuck-state class exists in the 'cancelled' handler (lines 375-387), though with lower likelihood since cancellation implies explicit user action.

## Impact
- Infinite loading spinner with no user-recoverable path
- Occurs when a user switches sessions during an error/cancel event processing
- Forces a full page reload to recover

## Suggestion
Add `loading.value = false` in the guard-fail paths:
```typescript
eventSource.addEventListener('error', (e) => {
  if (streamTimeout) { clearTimeout(streamTimeout); streamTimeout = null }
  if (!guard()) {
    loading.value = false  // Ensure loading is cleared even on guard fail
    return
  }
  disconnectStream()
  onLoadHistory().catch(() => {
    if (!guard()) {
      loading.value = false  // Clear loading on async guard fail
      return
    }
    // ... existing error handling
  })
  onStreamEnd?.('error')
})
```
