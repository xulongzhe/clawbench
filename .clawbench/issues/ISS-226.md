## Description
`DeriveEncryptionKey` (crypto.go:34-52) has a TOCTOU race between the `RLock` check (observing that derivation hasn't started) and the `Once` execution. While `sync.Once` guarantees only one goroutine executes the derivation, other goroutines that pass the `RLock` check before the `Once` completes may read an uninitialized or partially initialized encryption key.

## Impact
Concurrent API key operations (e.g., multiple chat sessions starting simultaneously on first run) could use an incomplete encryption key, producing ciphertext that cannot be decrypted later.

## Suggestion
Replace the RLock/check/RLock pattern with an RLock that waits for the `Once` to complete. Use a `sync.RWMutex` where the derivation goroutine holds a write lock, and readers acquire a read lock that blocks until the write lock is released. Alternatively, use a `channel` or `sync.WaitGroup` to signal completion.
