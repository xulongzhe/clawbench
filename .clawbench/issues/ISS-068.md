## Description
Cancelled/crashed execution paths use stale task.Status for UpdateTaskStats. The task status is read before execution and used after, but if the task was updated (e.g., paused, deleted) in the meantime, the stale status drives incorrect decisions — once/limited tasks keep firing even after being disabled.

## Impact
Tasks configured as "once" or "limited" continue to execute after they should have stopped. Wasted compute resources and unintended side effects from stale task runs.

## Suggestion
Read task status atomically within UpdateTaskStats, or re-read it under a lock after execution completes. Add a generation counter or version field to detect stale updates and discard them.
