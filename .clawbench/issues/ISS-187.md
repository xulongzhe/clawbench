## Description
The `HasRunningExecutions` check in the cron callback (scheduler.go:398) is not atomic with `runningExecutions.Store` (scheduler.go:571). Two concurrent cron ticks can both observe `HasRunningExecutions == false` and proceed to create two sessions for the same task.

## Impact
Defeats the single-execution-at-a-time invariant, causing double resource consumption and duplicate AI work. Both executions register in the sync.Map, so the second completion can incorrectly mark the task as completed.

## Suggestion
Use `sync.Map.LoadOrStore` keyed by taskID as a task-level lock before launching. If already present, skip. Delete in the defer. Alternatively, use `singleflight` keyed by taskID.
