## Description

`reconnect.reset()` is called unconditionally at the start of `connectStream()` at useChatStream.ts:211, which resets the retry counter before each connection attempt. If the SSE endpoint is persistently down, the counter never reaches `maxAttempts`, so the fallback to `pollUntilDone()` never triggers.

## Impact

If the backend SSE endpoint is persistently unreachable, the client never falls back to `pollUntilDone()`. The user sees a stuck streaming indicator with no content updates. This effectively creates an infinite retry loop that degrades the user experience without recovery.

## Suggestion

Move `reconnect.reset()` out of `connectStream()` and only reset on successful connection. The ideal place is on the first received SSE event or in the `EventSource.onopen` callback. This ensures the retry counter accurately tracks failed attempts and triggers the polling fallback when `maxAttempts` is exhausted.
