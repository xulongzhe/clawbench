## Description
connectStream called during active polling leaves an orphaned polling interval. If the user triggers a reconnect (e.g., by sending a message) while the HTTP polling fallback is active, `connectStream` opens a new SSE connection but does not clear the existing polling interval, leading to dual data streams.

## Impact
Duplicate events reaching the UI — messages appear twice, blocks are rendered in duplicate. Wasted network bandwidth from the unnecessary polling requests.

## Suggestion
In `connectStream`, explicitly clear any active polling interval before establishing the SSE connection. Use a single state variable to track the current connection mode and ensure only one mode is active at a time.
