## Description
The `serveTaskExecutions` function (scheduler.go:345-355) uses a LEFT JOIN on `chat_history` with conditions `ch.role = 'assistant' AND ch.streaming = 0 AND ch.deleted = 0`. When a session has multiple completed assistant messages (e.g., from AutoResumeBackend splits or multi-turn conversations), the JOIN produces one row per assistant message per execution, duplicating execution entries in the response.

## Impact
- Frontend renders ghost/duplicate execution entries
- With LIMIT clause, the row multiplication causes the limit to be consumed by duplicates, silently dropping real executions from the response
- The `content` field for each duplicated row contains a different assistant message, which is confusing

## Suggestion
Aggregate the JOIN to produce a single row per execution. Options:
1. Use `GROUP BY te.id` with `MAX(ch.content)` or `GROUP_CONCAT`
2. Use a subquery that selects a single assistant message per session (e.g., the latest finalized one)
3. Restructure the query to fetch executions first, then fetch content separately

Example:
```sql
SELECT te.id, te.session_id, te.trigger_type, te.status, te.created_at,
       te.read_at, te.summary,
       (SELECT ch.content FROM chat_history ch
        WHERE ch.session_id = te.session_id AND ch.role = 'assistant'
        AND ch.streaming = 0 AND ch.deleted = 0
        ORDER BY ch.created_at DESC LIMIT 1) AS assistant_content
FROM task_executions te
WHERE te.task_id = ?
ORDER BY te.created_at DESC
```
