## Description
Session token equals SHA-256(password+hardcoded_salt) — authenticating with the cookie is equivalent to knowing the password. The cookie IS a deterministic password derivative. If the cookie is leaked, the attacker can derive the original password. There is no way to invalidate a single session without changing the password for all sessions. No logout mechanism that actually revokes the token.

Additionally, the session cookie missing `Secure` flag means the token is transmitted over plaintext HTTP when TLS is not enabled.

Also, there is no logout/session revocation mechanism — no `/api/logout` endpoint.

## Impact
- Cookie leakage exposes the password itself (not just a revocable session)
- No session isolation — all sessions share the same token
- No way to revoke a compromised session
- Token transmitted in plaintext over HTTP without TLS

## Suggestion
1. Generate cryptographically random session tokens upon login
2. Store a server-side mapping of `token -> {username, expires}` in memory or SQLite
3. Set `Secure: true` on cookies when TLS is enabled
4. Implement `/api/logout` endpoint that clears cookie and removes server-side token
