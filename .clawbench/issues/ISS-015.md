## Description
`notifiedTaskCompletions` dedup key parsing uses naive `split('-')[0]` which is fragile, and eager cleanup on `runningCount === 0` causes potential re-notification. If a stale poll response arrives out of order, the notification fires again because the dedup key was already deleted.

## Impact
Duplicate task completion notifications under race conditions with polling.

## Suggestion
Use a more robust key format and a Map with explicit expiry timestamps. Don't delete dedup entries on runningCount===0 — only clean up for deleted task IDs.
